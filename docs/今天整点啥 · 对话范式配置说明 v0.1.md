# 今天整点啥 · 对话范式配置说明 v0.1

## 1. Event Mode 统计

根据当前项目文档与数据文件，正式实现和引用的 `eventMode` 一共有 4 种。

| 序号 | eventMode | 中文名 | 创建房间时含义 | 参与者问询重点 | 方案输出形态 |
| --- | --- | --- | --- | --- | --- |
| 1 | `meal_only` | 只吃饭 | 组织者初始判断这局主要解决吃饭 | 菜系、口味、忌口、预算、位置 | 单站餐厅方案，附备选餐厅 |
| 2 | `activity_only` | 只玩乐 | 组织者初始判断这局不以正餐为主 | 活动类型、体力强度、社交强度、预算、位置 | 单站娱乐方案，附备选活动 |
| 3 | `meal_activity` | 吃饭+娱乐 | 同一天既吃饭又安排饭后活动 | 第一站吃饭偏好、第二站娱乐偏好、两站距离、总预算 | 第一站吃饭 + 第二站娱乐 + 方案 B |
| 4 | `undecided` | 还没想好 | 组织者还没定大方向，希望让参与者共同提供方向 | 吃饭/玩乐/组合倾向、硬性雷区、预算、位置 | 决策 Agent 根据多数偏好决定最终方案类型 |

### 1.1 当前文档中的引用情况

- `TRD v0.2`：已定义 4 类 `eventMode`，并描述问询分支、动态升级/降级规则。
- `用户旅程与 AI 对话设计 v0.2`：已在状态机中体现 4 条分支。
- `Agent Skill 设计 v0.1`：已把 4 类 `eventMode` 写入 `Preference Agent` 的分支规则。
- `shenzhen_poi_seed_schema.v0.1.json` 与 `shenzhen_concrete_venue_seed.v0.1.json`：店铺适配字段 `suitableEventModes` 目前只需要覆盖 `meal_only`、`activity_only`、`meal_activity`。

### 1.2 为什么店铺数据不需要 `undecided`

`undecided` 是房间在决策前的状态，不是具体店铺或活动可以直接适配的类型。

处理方式：

```text
room.eventMode = undecided
  -> Preference Agent 收集参与者大方向倾向
  -> Decision Agent 决定 finalEventMode
  -> finalEventMode 只能落到 meal_only / activity_only / meal_activity
  -> 再用 finalEventMode 匹配店铺和活动
```

## 2. 用户模糊回答识别规则

### 2.1 技术选型结论

采用“整体语义理解为主，关键词匹配兜底”的混合方案。

不建议只用关键词匹配，也不建议完全依赖大模型自由判断。

| 方案 | 是否采用 | 原因 |
| --- | --- | --- |
| 纯关键词匹配 | 不作为主方案 | 简单稳定，但容易漏掉隐含表达，比如“别太折腾”“钱包顶不住”“我负责当好人” |
| 纯语义理解 | 不单独使用 | 表达理解强，但输出可能不稳定，需要字段约束和置信度 |
| 语义理解 + 关键词兜底 | 采用 | 既能理解自然表达，又能保证核心雷区、预算、位置等字段稳定落库 |

### 2.2 识别流程

```text
用户回答
  -> 规则层快速识别强信号关键词
  -> LLM 语义分类识别真实意图
  -> 生成 extractedPreferenceDelta
  -> 判断是否需要澄清
  -> 更新 ParticipantPreferencePackage 对应字段
```

### 2.3 规则层负责什么

规则层只负责识别高确定性的强信号，不负责复杂理解。

| 类型 | 示例关键词 | 结构化结果 |
| --- | --- | --- |
| 无偏好 | 随便、都行、听大家的、你看着办 | `no_preference` |
| 决策权让渡 | 看大家、我都可以、别问我 | `decisionStyle = follow_group` |
| 饮食硬限制 | 不吃辣、海鲜过敏、素食、不吃牛肉 | `hardConstraints` |
| 活动排除 | 不唱 K、不密室、怕高、不想暴晒 | `activityExclusions` |
| 预算敏感 | 别太贵、穷、50以内、AA不起 | `budget_sensitive` |
| 距离敏感 | 别太远、离我近点、不想跨城 | `distance_sensitive` |
| 局类型变更 | 光吃饭不够、加个 KTV、吃完找地方玩 | `eventModeChangeSuggestion` |

### 2.4 语义层负责什么

语义层负责理解用户没有直接说出口的偏好。

| 用户表达 | 语义理解 | 后台结果 |
| --- | --- | --- |
| 最近有点累，别搞太满 | 低体力、低折腾 | `energyLevel = low` |
| 想吃点正常的，别整活 | 安全牌、不踩雷 | `decisionStyle = guided_options_needed` |
| 不想吃辣，但也不知道吃啥 | 已排除辣，但菜系不明确 | `preferenceConflict = uncertain_cuisine_after_no_spicy` |
| 光吃饭感觉有点干 | 想增加饭后娱乐 | `eventModeChangeSuggestion = meal_activity` |
| 我家那边口味比较淡 | 偏清淡，可能关联家乡口味 | `tastePreference = light` |
| 我不想成为那个扫兴的人 | 有雷区但怕表达 | 低压追问硬限制 |

### 2.5 模糊回答分类

| 模糊类型 | 定义 | 处理方式 |
| --- | --- | --- |
| 无偏好型 | 用户真的不想参与决策 | 降低权重，只追问硬性雷区 |
| 排除型 | 用户说不想要什么，但没说想要什么 | 承认排除项，再给 3-4 个小选项 |
| 纠结型 | 用户想选，但怕选错 | 给推荐答案，降低选择压力 |
| 场景型 | 用户说感受，不说品类 | 转成体力、社交强度、预算、距离等字段 |
| 变更型 | 用户挑战当前局类型 | 记录为 `eventModeChangeSuggestion` |

### 2.6 是否需要追问

追问判断规则：

- 硬性雷区不清楚：允许追问 1 次。
- 菜系不清楚但用户已有排除项：允许追问 1 次。
- 用户连续两次表达“随便”：不继续追问，直接默认。
- 用户出现局类型变更：追问 1 次具体方向，比如 KTV、桌游、清吧。
- 用户明显不想回答：立刻跳过。

追问话术原则：

```text
不问“为什么”
不问“请详细说明”
改问“我帮你切一刀，哪个更接近”
每次给 3-4 个选项
必须允许“交给海鸥”
```

## 3. 家乡相关话题话术设计

### 3.1 设计目标

用户提到家乡时，Agent 不需要做复杂能力，只需要做到：

- 识别家乡话题可能对应口味偏好。
- 语气自然，不像查户口。
- 适当接梗，保持俏皮。
- 不强行刻板印象，不把某地用户直接绑定到某种口味。
- 最终仍要落到菜系、口味、雷区、预算等结构化字段。

### 3.2 触发场景

| 用户表达 | 识别方向 |
| --- | --- |
| 我是广东人，吃不了太辣 | 清淡、不辣、粤菜友好 |
| 我湖南的，可以吃辣 | 可接受辣、湘菜友好 |
| 我四川的，但今天不想吃辣 | 家乡不等于当前偏好，当前回答优先 |
| 想吃点家乡味 | 需要追问具体家乡或菜系 |
| 我东北的，想吃肉 | 肉食偏好、烧烤/东北菜/烤肉友好 |
| 我福建人，想清淡点 | 清淡、汤水、家常菜友好 |

### 3.3 通用回应模板

```text
海鸥收到，家乡味 buff 已加载。
不过本海鸥先不刻板印象，你今天是想吃家乡味，还是只是提醒我别踩雷？

A. 想吃点家乡味
B. 不一定家乡菜，但口味按我来
C. 只是告诉你雷区
D. 交给海鸥，别太离谱就行
```

### 3.4 不同家乡话题示例

| 用户输入 | 推荐回应 | 结构化倾向 |
| --- | --- | --- |
| 我广东人，吃不了辣 | 广东胃已接入，辣锅先下船。那我按清淡、不辣、粤菜/椰子鸡/早茶这类安全牌给你记？ | `tastePreference = light`, `hardConstraints = no_spicy` |
| 我湖南的，可以吃辣 | 湘味雷达响了。你是“能吃一点辣”，还是“辣度不够等于没吃”？ | `spicyTolerance = medium/high` |
| 我四川的，但今天不想吃辣 | 很好，今日川渝 DNA 选择休假。那我按不辣友好处理，不拿你的籍贯绑架你的胃。 | `tastePreference = not_spicy` |
| 我东北的，想吃肉 | 收到，肉食雷达开启。烤肉、火锅、东北菜这三条航线，你更想走哪条？ | `cuisinePreference = bbq/hotpot/northeast` |
| 我福建人，想清淡点 | 明白，今天走温和路线，不搞味觉蹦极。粤菜、汤锅、日料这种会更稳。 | `tastePreference = light` |
| 想吃家乡味 | 家乡味范围有点大，本海鸥先不乱飞。你家乡大概是哪边？或者直接说想吃什么菜。 | `needs_clarification = hometown_cuisine` |

### 3.5 禁止话术

- 不说“你是 XX 人所以一定爱吃 XX”。
- 不拿地域做冒犯性玩笑。
- 不对用户口味做优越/低级判断。
- 不让用户解释家乡文化背景。

推荐语气：

```text
可以玩梗，但不贴标签。
可以接话，但不查户口。
可以推荐，但以用户当前回答为准。
```

## 4. 每轮回答的结构化输出设计

### 4.1 沿用现有规范

不重新定义新的 handoff 规范，继续沿用当前 `Agent Skill设计 v0.1` 中的：

- `ParticipantPreferencePackage`
- `EventModeChangeSuggestion`
- `decisionStyle`
- `rawEvidence`
- `handoffSummary`

每一轮对话只做“增量结构化”，最后汇总进 `ParticipantPreferencePackage`。

### 4.2 每轮结构化处理流程

```text
用户回答
  -> 保存 rawAnswer
  -> 抽取 extractedPreferenceDelta
  -> 判断 ambiguitySignals
  -> 判断 eventModeChangeSuggestion
  -> 更新 conversationState
  -> 生成下一轮问题
```

### 4.3 每轮需要落的内容

| 字段 | 说明 | 写入最终包的位置 |
| --- | --- | --- |
| `questionKey` | 当前问题标识 | `rawEvidence` 的上下文，内部日志 |
| `rawAnswer` | 用户原话 | `rawEvidence` |
| `extractedPreferenceDelta` | 本轮抽取到的偏好增量 | `preference` |
| `hardConstraintsDelta` | 本轮新增硬限制 | `preference.hardConstraints` |
| `noPreferenceFieldsDelta` | 用户表示都行的字段 | `preference.noPreferenceFields` |
| `eventModeChangeSuggestionDelta` | 本轮是否提出局类型变更 | `eventModeChangeSuggestion` |
| `ambiguitySignals` | 模糊/纠结类型 | `decisionStyle` 与 `handoffSummary` |
| `confidence` | 本轮抽取置信度 | 最终 `confidence` 的参考 |

### 4.4 示例：用户回答“不吃辣，但不知道吃啥”

本轮用户可见回复：

```text
辣锅先踢下船。你不是没想法，是只排除了一个错误答案。

我帮你切一刀：
A. 粤菜/茶餐厅，稳妥不翻车
B. 日料/寿司，清淡少油
C. 猪肚鸡/椰子鸡，热乎但不辣
D. 交给海鸥，我只负责不吃辣
```

后台增量结构化：

```json
{
  "questionKey": "ASK_MEAL_PREFERENCE",
  "rawAnswer": "不吃辣，但不知道吃啥",
  "extractedPreferenceDelta": {
    "mealPreference": {
      "taste": ["not_spicy"],
      "cuisine": []
    },
    "hardConstraints": ["no_spicy"]
  },
  "ambiguitySignals": ["uncertain_cuisine_after_exclusion"],
  "decisionStyleDelta": "guided_options_needed",
  "needsClarification": true,
  "confidence": 0.86
}
```

最终 handoff 汇总时进入：

```json
{
  "preference": {
    "mealPreference": {
      "taste": ["not_spicy"],
      "cuisine": ["cantonese", "japanese", "coconut_chicken"]
    },
    "hardConstraints": ["no_spicy"]
  },
  "decisionStyle": "guided_options_needed",
  "rawEvidence": ["不吃辣，但不知道吃啥"],
  "handoffSummary": "用户不吃辣，但菜系不明确，需要决策 Agent 优先选择不辣友好餐厅。"
}
```

### 4.5 示例：用户回答“光吃饭不够，加个 KTV”

后台增量结构化：

```json
{
  "questionKey": "ASK_FINAL_CONFIRMATION",
  "rawAnswer": "光吃饭不够，加个 KTV",
  "extractedPreferenceDelta": {

      "preferred": ["ktv"]

  "eventModeChangeSuggestionDelta": {

    "evidence": ["光吃饭不够，加个 KTV"],
    "confidence": 0.9
  },
  "ambiguitySignals": [],
  "needsClarification": false,
  "confidence": 0.9
}
```

最终 handoff 汇总时进入：

```json
{
  "eventModeChangeSuggestion": {
    "fromMode": "meal_only",
    "suggestedMode": "meal_activity",
    "reason": "用户认为只吃饭不够，想增加饭后 KTV",
    "evidence": ["光吃饭不够，加个 KTV"],
    "confidence": 0.9
  },
  "rawEvidence": ["光吃饭不够，加个 KTV"],

}
```

## 5. 人格边界设定

### 5.1 沿用当前方案

人格边界不做大改，沿用之前确定的“码头海鸥，嘴碎但靠谱”。

### 5.2 可以做什么

- 可以俏皮吐槽选择困难。
- 可以用“码头”“海鸥”“薯条”“鸽子”“交卷”等产品梗。
- 可以轻微调侃“群里没人拍板”“大家都说随便”。
- 可以主动给推荐答案，帮用户降低决策压力。
- 可以在用户模糊时说“我帮你切一刀”。

### 5.3 不能做什么

- 不能连续追问，让用户感觉被审问。
- 不能使用地域、性别、职业等刻板印象。
- 不能为了玩梗忽略硬性禁忌。
- 不能编造用户没有表达的偏好。
- 不能把用户的家乡、位置、预算拿来冒犯式调侃。
- 不能输出“爹味建议”。

### 5.4 语气示例

推荐：

```text
收到，辣锅先踢下船。你不是没想法，是只排除了一个错误答案。
```

不推荐：

```text
你为什么不吃辣？请详细说明你的饮食习惯。
```

推荐：

```text
你负责随便，海鸥负责不让这局翻车。
```

不推荐：

```text
你这样回答会影响推荐准确性，请认真选择。
```

## 6. 当前结论

- 当前项目一共有 4 种 `eventMode`：`meal_only`、`activity_only`、`meal_activity`、`undecided`。
- 模糊回答识别采用混合方案：语义理解为主，关键词规则兜底。
- 家乡话题只作为口味线索，不做刻板绑定。
- 每轮回答采用增量结构化，最终汇总进现有 `ParticipantPreferencePackage`。
- 人格边界沿用“嘴碎但靠谱”，轻松但不冒犯，俏皮但不影响信息收集。
